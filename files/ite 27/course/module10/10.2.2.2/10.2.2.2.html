Troubleshooting OSPFv2 Routing Table Issues

In the topology in Figure 1, all of the routers have been configured to support OSPFv2 routing.

A quick look at the R1 routing table (Figure 2) reveals that it receives default route information, the R2 LAN (172.16.2.0/24) and the link between R2 and R3 (192.168.10.8/30). However, it does not receive the R3 LAN OSPFv2 route.

The output in Figure 3 verifies the OSPFv2 settings on R3. Notice that R3 only advertises the link between R3 and R2. It does not advertise the R3 LAN (192.168.1.0/24).

For an interface to be enabled for OSPFv2, a matching network command must be configured under the OSPFv2 routing process. The output in Figure 4 confirms that the R3 LAN is not advertised in OSPFv2.

The example in Figure 5 adds a network command for the R3 LAN. R3 should now advertise the R3 LAN to its OSPFv2 neighbors.

The output in Figure 6 verifies that the R3 LAN is now in the routing table of R1.
